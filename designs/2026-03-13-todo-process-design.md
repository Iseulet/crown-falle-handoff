
## 5. 항목 등록 프로세스 (언제, 누가, 어떻게)

### 5-1. Desktop 채팅 중 발견

```
원주가 채팅 중 "이것도 해야 해" 발언
  → Claude Desktop이 즉시 기록:
    "TODO 항목 추가: {내용} [데스크탑:MM-DD]"
  → 다음 CLI 세션에서 TODO.md에 실제 기입
```

**Desktop에서는 TODO를 직접 수정할 수 없으므로**, 채팅 내에서 항목을 명시적으로 선언하고 CLI에서 반영한다. 메모리에도 미반영 TODO가 있음을 기록한다.

### 5-2. CLI 구현 중 발견

```
@Implementor가 구현 중 "이것도 고쳐야겠다" 인지
  → 즉시 TODO.md에 기입 (작업 흐름 중단하지 않음)
  → 5단계 완료 체크리스트에 "TODO.md 갱신" 항목 추가
```

### 5-3. 설계 문서 확정 시

```
설계 문서의 "미결정/후순위" 섹션이 있으면
  → @Planner가 각 항목을 TODO.md 🟢 백로그에 복사
  → 출처 태그: [설계:{문서명}]
```

### 5-4. 코드 리뷰 후

```
@Reviewer가 "수정 권고" 항목 중 즉시 수정하지 않는 것
  → TODO.md 🟡 다음에 기입
  → 출처 태그: [리뷰:MM-DD]
```

### 5-5. Godot 테스트 후

```
원주가 테스트 중 발견한 이슈/개선점
  → CLI에서 TODO.md에 기입
  → 출처 태그: [테스트:MM-DD]
```

## 6. 항목 승격/강등 규칙

```
🟢 백로그 → 🟡 다음:
  - 해당 Phase 시작 시 관련 항목 일괄 승격
  - 원주가 명시적으로 "이거 다음에 하자"

🟡 다음 → 🔴 긴급:
  - 현재 작업이 이 항목 없이 진행 불가
  - 버그로 인해 테스트 불가

🟡 다음 / 🔴 긴급 → ✅ 완료:
  - 구현 + 커밋 완료 시
  - 항목을 ✅ 섹션으로 이동 + 완료일 기입

✅ 완료 → 아카이브:
  - 월 초에 지난달 완료 항목 일괄 삭제 (git log에 이력 남음)
```

## 7. 세션 시작/종료 시 TODO 루틴

### 세션 시작 (CLI)

```

  1. CURRENT.md 읽기
  2. SCRATCHPAD.md 읽기

추가:
  3. TODO.md 🔴 긴급 섹션 확인 → 있으면 우선 처리
  4. TODO.md 🟡 다음 섹션 확인 → 작업 계획 수립 참고
```

### 세션 종료 (CLI)

```
기존 5단계 완료 체크리스트에 추가:

     - 이번 세션에서 발견한 새 항목 기입했는가?
     - 완료한 항목 ✅로 이동했는가?
```

### Desktop 세션 시작

```
기존:
  프로젝트 나레지 또는 sync 파일로 컨텍스트 파악

추가:
  TODO.md도 확인 가능 (handoff 리포 싱크 시 포함)
```

### 소스 → TODO.md 매핑

| 소스 | 처리 |
|------|------|
| `project-context.md` 미완료 체크박스 | 🟡 다음으로 이동. context에서는 삭제하지 않음 (이중 관리 허용, context는 "현재 Phase" 뷰) |
| `SCRATCHPAD.md` Session Log 내 할 일 | 🟢 백로그로 이동. SCRATCHPAD에서는 삭제 (학습 메모만 남김) |
| 설계 문서 "미결정/후순위" | 🟢 백로그로 복사. 설계 문서 원본은 유지 (설계 맥락 보존) |
| 메모리 내 미반영 항목 | 🟡 다음으로 기입 |

## 9. project-context.md와의 관계

```
project-context.md:
  "현재 Phase에서 뭘 하고 있는가" — 진행 상황 스냅샷
  미완료 체크박스 = 현재 작업 범위 내 남은 것

TODO.md:
  "전체적으로 뭘 해야 하는가" — 크로스-Phase 백로그
  모든 우선순위의 할 일 통합 뷰

관계:
  context의 미완료 항목 ⊂ TODO.md
  TODO.md ⊃ context + 설계 후순위 + 세션 중 발견 + 테스트 이슈
```

겹치는 항목은 허용한다. context는 "지금 여기" 뷰이고, TODO는 "전체 뷰"이므로 역할이 다르다.
