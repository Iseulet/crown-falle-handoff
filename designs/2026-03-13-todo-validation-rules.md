# Part 1: 중복 이슈 체크

## 1-1. 문제

```
시나리오 A — 등록 시점 중복:
  3월 12일: /todo HP_BAR_Y 높이 조정 필요
  3월 14일: /todo HP 바 위치가 너무 높음
  → 같은 이슈인데 표현이 달라서 2개 등록

시나리오 B — 처리 시점 중복:
  🟡 투사체 FX Trail 구현
  🟢 투사체 GPUParticles3D FX 연결
  → /todo done 처리 시 하나만 완료되고 나머지가 잔류

시나리오 C — 범위 포함 중복:
  🟡 Phase B 전체 검증
  🟡 투사체 비행 테스트
  🟡 LOS 동작 확인
  → 상위 항목이 하위를 포함하는데 개별 등록
```

## 1-2. 등록 시 중복 감지

`/todo` 또는 `TODO:` 등록 시, Claude Code가 기존 항목과 유사도를 체크한다.

### 감지 방식

```
등록 요청: "/todo HP 바 위치가 너무 높음"

Claude Code 처리:
  1. TODO.md 전체 항목 스캔 (🔴 + 🟡 + 🟢)
  2. 키워드 매칭: "HP" "바" "위치" "높"
  3. 유사 항목 발견 시 → 확인 요청

출력:
  ⚠️ 유사 항목이 있습니다:
    🟡 #2. HP_BAR_Y=1.8 높이 최종 조정 `[구현:03-12]`

  어떻게 할까요?
    (1) 그래도 새로 등록
    (2) 기존 항목에 메모 추가
    (3) 취소
```

### 매칭 기준

단순 키워드 겹침으로 판단한다. 정교한 NLP가 아니라, **핵심 명사 2개 이상 겹침**이면 유사 후보로 제시.

```
매칭 대상 추출:
  "HP 바 위치가 너무 높음" → 핵심어: [HP, 바, 위치, 높]
  기존 "HP_BAR_Y=1.8 높이 최종 조정" → 핵심어: [HP, BAR, 높이, 조정]

  겹침: HP, 높 → 2개 이상 → 유사 후보
```

**정확한 중복이 아닌 "유사 후보 제시"**가 목적. 최종 판단은 원주가 한다.

### 자동 무시 케이스

- 기존 항목이 ✅ 완료 섹션에만 있으면 → 유사 경고 없이 등록 (다시 발생한 이슈)
- 기존 항목과 출처 태그가 다르면 → 경고는 하되 "(같은 이슈를 다른 맥락에서 발견)" 표시

## 1-3. 완료 처리 시 연쇄 체크

`/todo done` 실행 시, 완료된 항목과 유사한 나머지 항목을 자동 스캔한다.

```
/todo done HP_BAR

Claude Code:
  ✅ 완료 처리: HP_BAR_Y=1.8 높이 최종 조정

  📎 유사 항목이 남아있습니다:
    🟡 #8. HP 바 위치가 너무 높음 `[테스트:03-14]`

  함께 완료 처리할까요? (y/n)
```

원주가 `y`하면 일괄 완료. `n`이면 개별 유지.

## 1-4. 범위 포함 관계 감지

상위 항목이 하위 항목들을 포함하는 경우, 상위 완료 시 하위도 제안.

```
/todo done Phase B 전체 검증

Claude Code:
  ✅ 완료 처리: Phase B 전체 검증

  📎 하위 범위로 보이는 항목:
    🟡 #5. 투사체 비행 테스트
    🟡 #6. LOS 동작 확인
    🟡 #7. delivery 분기 테스트

  이 항목들도 함께 완료 처리할까요?
    (1) 전부 완료
    (2) 개별 선택
    (3) 유지 (아직 안 끝남)
```

# Part 2: 개발 방향 변경에 의한 무효화

## 2-1. 문제

```
시나리오 A — 설계 변경:
  기존 TODO: "ARM/RES % 감소 밸런스 조정"
  설계 변경: ARM/RES가 게이지형으로 전환됨
  → 기존 TODO가 의미 없어짐 (% 방식 자체가 폐기)

시나리오 B — 상위 시스템 대체:
  기존 TODO: "반격 시스템 구현"
  설계 변경: 교전 v3에서 반격을 제거하고 스킬로 이관
  → TODO 방식이 근본적으로 바뀜

시나리오 C — 우선순위 역전:
  기존 TODO: 🟢 "AoE 마법 설계" (백로그)
  설계 변경: Phase C가 스킬 확장으로 결정
  → 갑자기 🟡 또는 🔴로 승격 필요
```

## 2-2. 무효화 감지 시점

| 시점 | 트리거 | 체크 방식 |
|------|--------|----------|
| **설계 문서 확정 시** | @Planner가 신규 설계를 확정 | 설계의 "변경하지 않는 항목" + "폐기 항목"을 TODO와 대조 |
| **Phase 전환 점검** | `/todo review` 실행 | 전체 항목 × 새 Phase 방향 정합성 체크 |
| **확정 결정사항 변경** | DECISIONS.md 갱신 | 영향받는 TODO 항목 플래그 |

## 2-3. 설계 문서 확정 시 자동 대조

새 설계 문서가 확정될 때, Claude Code가 TODO.md를 스캔하여 영향받는 항목을 보고한다.

### 트리거 조건

```
원주: "이 설계 확정할게" (설계 문서 승인)
  │
  ▼
Claude Code:
  1. 설계 문서의 핵심 변경 키워드 추출
     예: "ARM/RES 게이지형 전환", "반격 제거", "delivery 3분기"
  2. TODO.md 전체 스캔 → 키워드 매칭
  3. 매칭된 항목에 대해 영향도 분류
```

### 출력

```
📋 설계 확정에 따른 TODO 영향 분석

확정: stat-system-revision.md (ARM/RES 게이지형 전환)

⚠️ 무효화 가능:
  🟢 #12. ARM/RES % 밸런스 조정 — % 방식 폐기됨
    → 삭제 또는 "ARM/RES 게이지 밸런스 조정"으로 수정?

🔄 방식 변경 필요:
  🟡 #4. T1+T2 전체 검증 — 게이지 흡수 로직이 추가되어 검증 범위 확대
    → 메모 추가: "(ARM/RES 게이지 흡수 포함)"

✅ 해결됨:
  🟢 #15. ARM 상한 80% 적정한지 확인 — 게이지형은 상한 개념 없음
    → 완료 처리?

영향 없음: 나머지 19개

어떻게 처리할까요?
```

## 2-4. DECISIONS.md 변경 시 플래그

확정 결정사항이 변경/추가될 때, 관련 TODO에 `⚠️ STALE` 플래그를 붙인다.

```markdown
# TODO.md 내 플래그 표기

- [ ] ⚠️ ARM/RES % 밸런스 조정 `[설계:stat-system]` `[2026-03-08]`
       ↑ STALE 플래그 — 설계 변경으로 유효성 미확인
```

### 플래그 해제 조건

- 원주가 `/todo review`에서 확인 후 수정/삭제/유지 결정
- 또는 `/todo update {키워드} {새 내용}`으로 직접 수정

## 2-5. /todo review에 무효화 체크 통합

기존 `/todo review` (Phase 전환 점검)에 무효화 분석을 추가한다.

```
/todo review 출력에 추가되는 섹션:

🔍 유효성 검증:

  ⚠️ STALE (설계 변경 영향):
    🟢 #12. ARM/RES % 밸런스 조정
      → 근거: stat-system-revision.md (2026-03-13)
      → 제안: 삭제 또는 "게이지 밸런스 조정"으로 수정

  ⏰ 장기 체류 (14일+):
    🟡 #1. FBX 클립 이름 확인 — 등록 2026-03-12 (2일)
    🟢 #18. dodge.glb 수급 — 등록 2026-03-12 (2일)
    → 아직 짧음, 7일 이상 시 하이라이트

  ✅ 이미 해결된 것 같은 항목:
    🟡 #3. 싱크 파일 갱신 — 오늘 싱크 실행됨
      → 완료 처리?
```

## 2-6. /todo update 커맨드 추가

기존 항목의 내용을 수정할 수 있는 커맨드.

```
/todo update ARM 밸런스 → ARM/RES 게이지 초기값 밸런스 테스트

Claude Code:
  ✏️ 수정:
    이전: ARM/RES % 밸런스 조정
    변경: ARM/RES 게이지 초기값 밸런스 테스트
  ⚠️ STALE 플래그 해제.
```

# Part 3: CLAUDE.md 추가 규칙 (통합)

```markdown
## TODO 검증

### 등록 시 중복 감지
- 신규 등록 시 기존 항목과 핵심 명사 2개+ 겹침 → 유사 후보 제시
- ✅ 완료 항목과만 겹치면 경고 없이 등록
- 최종 판단은 사용자 (자동 차단 아님)

### 완료 시 연쇄 체크
- /todo done 시 유사 항목 자동 스캔 → 함께 완료 제안
- 상위 범위 항목 완료 시 하위 항목도 제안

### 설계 변경 무효화
- 설계 문서 확정 시 TODO 영향 분석 자동 실행
- 영향받는 항목에 ⚠️ STALE 플래그
- /todo review에서 STALE 항목 일괄 정리
- /todo update {키워드} → {새 내용} 으로 수정 + 플래그 해제
```

# 커맨드 전체 목록 (최종)

| 커맨드 | 동작 |
|--------|------|
| `/todo {내용}` | 🟡에 등록 (중복 감지 포함) |
| `TODO: {내용}` | 🟡에 등록 (자연어, 중복 감지 포함) |
| `/todo list` | 🔴 + 🟡 출력 |
| `/todo all` | 전체 출력 |
| `/todo done {키워드}` | ✅로 이동 (연쇄 체크 포함) |
| `/todo urgent {키워드}` | 🟡 → 🔴 |
| `/todo backlog {키워드}` | 🟡 → 🟢 |
| `/todo update {키워드} → {새 내용}` | 항목 수정 + STALE 플래그 해제 |
| `/todo review` | Phase 전환 점검 (분류 + 무효화 + 체류 분석) |
| `/todo milestone` | 마일스톤 점검 (전체 분포 + 방향 제안) |
